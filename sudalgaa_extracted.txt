Бакалаврын судалгааны ажлын үндэслэл
Удиртгал
Орчин үед программ хангамжийн системүүдийн хөгжлийн чиг хандлага нь бизнесийн орчны түргэн өөрчлөлт, хэрэглэгчийн хэрэгцээний олон янз байдал, үйлчилгээний тасралтгүй сайжруулалтын шаардлага зэргээс шалтгаалан уян хатан, өргөтгөх боломжтой архитектурын шийдлүүдийг эрэлхийлэх болсон. Үүлэн тооцоолол, контейнерчлэл, автомат байршуулалт зэрэг практикууд өргөн хэрэглэгдэх болсноор архитектурын дизайн нь системийн амьдралын мөчлөгийн бүх шатанд нөлөөлөх стратегийн хүчин зүйл болжээ.
Уламжлалт монолит архитектур нь хөгжүүлэлтийн эхэн үед энгийн боловч систем төвөгшсөн үед өөрчлөлтийн өртөг өсч, масштаблалт хязгаарлагддаг. Харин микросервис архитектур нь бизнесийн чадамжаар чиглэсэн жижиг, бие даасан сервисүүдэд тулгуурлан уян хатан байдал, сонгон өргөтгөх боломжийг олгодог ч түгээгдсэн системийн нарийн төвөгийг нэмэгдүүлдэг. Иймээс архитектурын сонголтыг гүйцэтгэл, найдвартай байдал, өргөтгөх боломж зэрэг чанарын шинжүүдээр бодитой харьцуулан дүгнэх шаардлагатай байна.
Энэхүү дипломын ажлын зорилго нь монолит, SOA, микросервис архитектуруудын харьцуулсан дүн шинжилгээ хийж, микросервис архитектурт суурилсан уулзалт/эвент төлөвлөлтийн системийг хэрэгжүүлэн турших явдал юм. Системийн туршилтын кейс болгон Монгол Улсад зохион байгуулагдаж буй төрөл бүрийн арга хэмжээг нэг дор төвлөрүүлэн харуулах эвент платформыг сонгосон бөгөөд энэхүү домэйн нь хэрэглэгчийн оргил ачаалал, захиалгын зэрэгцээ ажиллагаа, мэдэгдлийн урсгал зэрэг олон төрлийн функциональ шаардлагыг агуулж, архитектурын ялгааг гүйцэтгэлийн үзүүлэлтээр бодитоор хэмжих боломжийг олгодог.
Системийн танилцуулга
Энэхүү дипломын ажлын хүрээнд хэрэгжүүлэх систем нь Монгол Улсад зохион байгуулагдаж буй төрөл бүрийн арга хэмжээг нэг дор төвлөрүүлэн харуулах уулзалт болон эвент төлөвлөлтийн платформ вэб аппликейшн юм. Одоогийн байдлаар Монгол Улсын хэмжээнд зохион байгуулагдаж буй соёл, спорт, бизнес, боловсролын арга хэмжээнүүдийн талаарх мэдээлэл олон янзын эх сурвалжаар (сошиал медиа, зар сурталчилгаа, байгууллагын вэбсайт) тархсан, төвлөрсөн нэгдсэн платформ байхгүйгээс хэрэглэгчид сонирхсон эвентээ олж, мэдээлэл авахад хүндрэлтэй байдаг. Мөн эвент зохион байгуулагчид өөрсдийн арга хэмжээг сурталчлах, оролцогчдын бүртгэлийг удирдах үр дүнтэй механизмгүй байна.
Төвлөрсөн эвент платформ нь эдгээр асуудлыг шийдвэрлэхэд чиглэгдсэн бөгөөд дараах үндсэн боломжуудыг хэрэглэгчдэд олгоно:
Эвент хайх, нээх - төрөл, огноо, байршил зэрэг шалгуураар эвентүүдийг хайх, дэлгэрэнгүй мэдээлэл харах
Эвентэд бүртгүүлэх - сонирхсон эвентээ хадгалах, бүртгүүлэх, бүртгэлээ цуцлах
Эвент үүсгэх, удирдах - шинэ эвент үүсгэх, мэдээлэл засварлах, зураг оруулах, оролцогчдын жагсаалт харах
Хэрэглэгчийн бүртгэл - нэвтрэх эрх үүсгэх, хувийн мэдээлэл удирдах, нэвтрэлтийн хамгаалалт
Систем нь микросервис архитектур дээр суурилсан бөгөөд 3-4 бие даасан сервис (User Service, Event Service, Booking Service, Notification Service) хэлбэрээр хэрэгжинэ. Сервис бүр нь тусдаа өгөгдлийн сантай, бие даан байрших боломжтой бөгөөд хоорондоо REST API-аар харилцах болно. Энэхүү архитектур нь системийн уян хатан байдал, өргөтгөх боломж, алдаа тусгаарлалтыг хангахаас гадна монолит, SOA архитектуруудтай харьцуулан судлах боломжийг бүрдүүлнэ.
Платформыг хэрэгжүүлснээр Монгол Улсын эвент зохион байгуулалтын практикт бодит хувь нэмэр оруулж, хэрэглэгчдэд эвентүүдийн талаарх мэдээллийг нэгдсэн цэгээс авах, зохион байгуулагчдад үр дүнтэй бүртгэл, сурталчилгааны механизм бий болгох боломж олгоно.
2. Бакалаврын судалгааны ажлын зорилго, зорилтууд
1) Бакалаврын судалгааны ажлын зорилго
Monolithic, SOA, Microservices архитектуруудын онолын болон хэрэгжилтийн ялгааг харьцуулан шинжилж, microservices архитектур дээр суурилсан уулзалт төлөвлөлтийн системийг хөгжүүлэн, гүйцэтгэл ба чанарын үзүүлэлтээр үнэлж дүгнэхэд оршино.
2) Бакалаврын судалгааны ажлын зорилтууд
Архитектурын онолын судалгаа хийж, monolithic, SOA, microservices-ийн үндсэн ойлголт, зарчим, чанарын шинжүүдэд нөлөөлөх хүчин зүйлсийг системтэй нэгтгэх.
Архитектуруудын харьцуулалт хийж, deployability, scalability, maintainability, reliability зэрэг шалгуураар давуу/сул талыг тодорхойлох.
Уулзалт, эвент төлөвлөлтийн системийн бизнес домэйн, хэрэглэгчийн хэрэгцээ, функциональ ба функциональ бус шаардлагыг тодорхойлох.
Microservices архитектурын загвар боловсруулж, bounded context-д суурилсан сервис задлал, өгөгдлийн удирдлага, харилцааны хэлбэр (sync/async)-ийг төлөвлөх.
Төлөвлөсөн архитектурын дагуу системийг хэрэгжүүлж, API gateway, service discovery зэрэг дэд бүтцийн шийдлүүдийг интеграцлах.
Гүйцэтгэлийн туршилт хийж, latency, throughput, error rate, нөөцийн хэрэглээ зэрэг үзүүлэлтээр архитектурын нөлөөг хэмжих.
Туршилтын үр дүнг задлан шинжилж, архитектур сонголтын нөхцөл–зөвлөмж боловсруулан, дүгнэлт гаргах.
3) Бакалаврын судалгааны ажлын шинэлэг тал
Энэхүү ажил нь нэг домэйн (уулзалт/эвент төлөвлөлт)-ийн хүрээнд monolithic–SOA–microservices архитектуруудыг чанарын шинжүүдээр системтэй харьцуулж, microservices хэрэгжилтийн бодит жишээ систем дээр туршилтаар баталгаажуулахад чиглэж буйгаараа онцлог. Мөн Meeting-ийн microservices хөрвүүлэлтийн кейс судалгаанаас зүйн сургамж авч, бакалавр/дипломын түвшинд хэрэгжихүйц загвар, хэмжилтийн хүрээг тодорхойлохыг зорьж байна.
4) Судалгааны ажлын ач холбогдол
Онолын ач холбогдол. Архитектурын хэв маягуудын (monolithic, SOA, microservices) ялгааг нэг домэйн дээр нэгдсэн шалгуур, хэмжүүрээр тайлбарлан, архитектур сонголтын шийдвэр гаргалтад хэрэглэгдэх дүгнэлт өгнө.
Практик ач холбогдол. Microservices загвараар уулзалт төлөвлөлтийн системийн бодит хэрэгжилт, сервисийн задлал, дэд бүтцийн компонентууд болон гүйцэтгэлийн туршилтын жишээг санал болгож, ижил төрлийн систем хөгжүүлэлтэд дахин ашиглах боломж бүрдүүлнэ
3.Программ хангамжийн архитектурын онолууд
1) Monolithic архитектур
Monolithic архитектур нь нэг аппликейшний бүх бизнес логик, өгөгдлийн хандалт, интерфейсийн давхаргууд нэг кодын сан болон нэг байршуулалтын нэгжид төвлөрсөн зохион байгуулалт юм. Ийм бүтэц нь хөгжүүлэлтийн эхэн үед энгийн, хөгжүүлэлтийн орчин тохируулахад хялбар байдаг тул өргөн хэрэглэгддэг.
Үндсэн зарчмууд.
Нэгтгэсэн кодын сан, нэг artifact (jar/war гэх мэт)
Нэгдсэн өгөгдлийн сан (ихэвчлэн shared DB)
Дотоод дуудлага нь ихэвчлэн function call түвшинд (in-process), network overhead бага
Release/deploy нь бүхэл системээрээ зэрэг хийгддэг (single deployment unit)
Давуу тал.
Архитектурын төвөгшил, хөгжүүлэлтийн босго харьцангуй бага.
Дотоод дуудлага хурдан (network latency байхгүй), гүйцэтгэлийн тодорхой нөхцөлд ашигтай.
Debugging, локал туршилт хийхэд хялбар (нэг runtime орчин).
Сул тал.
Жижиг өөрчлөлтөд ч бүхэл системийг дахин багцалж байрлуулах шаардлагатай болдог нь релизийн эрсдэлийг өсгөнө.
Модуль хоорондын хамаарал ихсэж, засвар үйлчилгээ хүндрэх, техникийн өр нэмэгдэх хандлага үүснэ.
Масштаблалт нь бүхэл системийг хамтад нь өргөтгөхөд хүрч, нөөцийн үр ашиг муудна.
Ашиглагддаг орчин. Стартапын эхний хувилбар, жижиг/дунд хэмжээний дотоод систем, домэйн нь тогтвортой, багийн хэмжээ жижиг үед илүү тохиромжтой.
2) Service-Oriented Architecture (SOA)
SOA нь байгууллагын түвшний олон төрлийн системүүдийг “үйлчилгээ” (service) хэлбэрээр задлан, стандартчилсан service contract ба интерфейсээр дамжуулан харилцуулах, дахин ашиглах, найруулах (composition/orchestration) боломж олгодог архитектурын парадигм гэж тодорхойлогдоно.
Үндсэн зарчмууд (Erl-ийн зарчмуудаар).
Standardized Service Contract: үйлчилгээний боломж, өгөгдлийн загвар, бодлогыг гэрээгээр тогтоох.
Loose Coupling: хэрэгжүүлэлтээс хамаарал багатай уялдаа.
Abstraction: дотоод хэрэгжилтийг нуух.
Reusability & Composability: бизнес процессийг үйлчилгээний найруулгаар бүтээх.
Autonomy & Statelessness: бие даасан гүйцэтгэл, масштаблалт дэмжих.
Давуу тал.
Байгууллагын олон системийн интеграц, дахин ашиглалт сайн.
Гэрээт харилцаа нь өөр баг/системүүдийн уялдааг зохион байгуулалттай болгодог.
Сул тал.
Governance, ESB төвтэй хэрэгжилтэд төвлөрөл бий болж, өөрчлөлтийн хурд саарах эрсдэлтэй.
Service оркестраци/найруулга хэт төвлөрөхөд single point of failure үүсэх магадлал нэмэгддэг (судлаачид энэ эрсдэлийг governance-ийн түвшинд бууруулах шаардлагатай гэж үздэг).
Ашиглагддаг орчин. Том байгууллагын интеграц (legacy систем олон), ERP/CRM зэрэг олон системийг бизнес процессоор холбох шаардлагатай нөхцөл.
3) Microservices архитектур
Microservices нь нэг аппликейшнийг жижиг сервисүүдийн багц болгон хөгжүүлж, сервис бүр өөрийн процессод ажиллан, хөнгөн протоколоор харилцаж, бүрэн автоматжсан байршуулалтаар independently deployable байхыг зорьдог архитектурын хэв маяг юм.
Үндсэн зарчмууд.
Business capability / bounded contextд суурилсан сервис задлал.
Decentralized data management (database per service гэх мэт).
Automation & DevOps: CI/CD, containerization.
Resilience & Observability: fault isolation, metrics/logs/tracing.
Lightweight communication: REST/gRPC, event-driven messaging.
Давуу тал.
Бие даасан deploy → релизийн эрсдэл буурч, өөрчлөлтийн хурд нэмэгддэг.
Сонгон масштаблах боломж → өндөр ачаалалтай сервисийг тусад нь өргөтгөнө.
Fault isolation → нэг сервисийн алдаа бүхэл системд тархах магадлалыг бууруулна.
Сул тал.
Distributed system-ийн төвөгшил: network latency, partial failure, consistency, monitoring, tracing нэмэгдэнэ.
Өгөгдлийн бүрэн бүтэн байдал (ACID) нь олон сервис дамжих үед хүндрэх ба saga зэрэг шийдэл шаардана.
Ашиглагддаг орчин. Том/өсөлттэй бүтээгдэхүүн, олон баг зэрэг хөгжүүлдэг, ачаалал хэлбэлзэлтэй, өндөр бэлэн байдал шаарддаг үйлчилгээ.

1.1.1 Микросервис гэж юу вэ?
Мартин Фаулер болон Жеймс Льюис нар микросервис архитектурыг нэг аппликейшнийг “жижиг сервисүүдийн цогц” болгон хөгжүүлэх хэв маяг гэж тодорхойлж, сервис тус бүр өөрийн процесст ажиллан, хөнгөн механизм (ихэвчлэн HTTP resource API)-аар харилцаж, бизнес чадамжийн хүрээнд бүтээгдэж, бүрэн автоматжуулсан байршуулалтын механизмаар бие даан байршуулдаг болохыг онцолсон байдаг.
Тухайн тодорхойлолтын гол санаа нь “жижиг код” бус, өөрчлөлт/байршуулалт/масштабын тусгаар тогтнол юм. Үүнийг орчин үеийн микросервисийн паттернүүдийн сан (microservices.io) мөн “suite of services” ба сервис тус бүрийн автономит байдлаар баталгаажуулан тайлбарладаг.

DDD ба домэйн задлал
Domain-Driven Design ба Bounded Context
DDD-ийн суурь санаа нь: программын бүтэц нь техникийн давхарга (UI/DB) биш, харин домэйн мэдлэг ба бизнесийн хэл дээр төвлөрөх ёстой. DDD Reference-д “ubiquitous language”-ийг explicit bounded context дотор тогтоон ашиглахыг голчлон тодорхойлдог.
Микросервисийн тухай Microsoft-ийн архитектурын гарын авлагад: микросервис архитектур нь “жижиг автономит сервисүүдийн цуглуулга” бөгөөд сервис тус бүр тодорхой бизнес чадамжийг bounded context дотор хэрэгжүүлэх ёстой гэж шууд заасан байдаг.
Ингэснээр DDD-ийн bounded context нь микросервисийн boundary тогтоох “онолын шүүлтүүр” болж өгч, “хаанаас хамаарал эхэлж, хаана автономит байдал дуусах вэ?” гэсэн асуудлыг шийдвэрлэхэд тусалдаг.
Уулзалт, эвент төлөвлөлтийн домэйнд bounded context-оор задлах жишээ
Дипломын системийн зорилго (уулзалт/эвент төлөвлөлт, суудал захиалга,мэдэгдэл) нь нэг кодын бааз дотор “модуль” байдлаар амархан эхэлж болох ч үйлдвэрлэлийн шаардлага нэмэгдэхийн хэрээр бизнес дүрэм (rules) ба өгөгдлийн эзэмшил өөр өөр ритмээр өөрчлөгдөх магадлал өндөр.
Доорх нь домэйн хэллэг дээр суурилсан практик decomposition жишээ (дипломын архитекторын зохиомж хийхэд шууд хэрэглэж болох):
Identity & Access Context: нэвтрэлт, токен, эрх, session
User Profile Context: хэрэглэгчийн профайл, тохиргоо, байгууллага/баг
Event Management Context: эвент үүсгэх, хөтөлбөр, тасалбарын төрөл, оролцогч
Venue & Seating Context: талбай/танхим, суудлын зураглал, багтаамж, суудлын түгжих дүрэм
Booking & Order Context: захиалга үүсгэх/баталгаажуулах, төлөв (pending/confirmed/canceled)
Payment Context (эсвэл интеграц): төлбөрийн баталгаажуулалт/буцаалт, чек/лог
Notification Context: email/SMS/push, мэдэгдэл илгээх, retry, template
Search/Read Context (шаардлагатай бол): эвент хайлт, жагсаалт, тайлангийн read-model (CQRS хэрэгжүүлэх боломж).
Ийм decomposition нь сервис тус бүрийг “single business capability / bounded context” хүрээнд барих Microsoft-ийн зөвлөмжтэй нийцнэ.

Харилцаа, өгөгдөл, үйл ажиллагаа ба загварууд
Харилцааны хэв маяг
1.4.1 REST (HTTP resource API)
REST нь интернет-цар хүрээний системийн архитектурын зарчим (constraints)-уудыг тайлбарласан Architectural Styles and the Design of Network-based Software Architectures, Fielding 2000 dissertation-д суурилдаг бөгөөд өнөөгийн веб API загварчлалын онолын суурь хэвээр байна.
Микросервисийн практикт REST нь “CRUD-т төвлөрсөн” болон client-facing API-д ойлгомжтой, HTTP экосистемтэй сайн нийцдэг ч, дотоод сервис хоорондын өндөр давтамжтай дуудлагад latency/overhead үүсгэх эрсдэлтэй.
1.4.2 gRPC (RPC over HTTP/2 + Protobuf)
gRPC, rpc framework нь албан баримт бичигтээ Protocol Buffers-ийг IDL болон message format болгон ашиглаж болохыг тайлбарладаг.
Мөн gRPC-ийн HTTP/2 дээр “carried over HTTP/2 framing” хэлбэрээр тодорхойлогдсон протоколын тайлбар GitHub дээр бий.
Protocol Buffers, data serialization format нь “language-neutral, platform-neutral” бүтэцтэй өгөгдөл сериалчлах механизм бөгөөд жижиг/хурдан гэж тайлбарлагддаг.
Иймээс gRPC нь сервис хоорондын олон дуудлагатай, type-safe contract шаарддаг, streaming хэрэгтэй тохиолдолд давуу талтай.
1.4.3 Message broker (асинхрон харилцаа)
RabbitMQ, message broker нь AMQP 0-9-1 загварт “exchange → queue” чиглэлээр мессежийг чиглүүлэх (routing) ойлголтыг тодорхойлж, exchange нь publisher-ээс ирсэн мессежийг queue/stream рүү “route” хийдэг гэж тайлбарладаг.
Apache Kafka, event streaming platform-ийн албан танилцуулгад Kafka нь кластер дээр ажиллаж, topic гэж нэрлэгдэх урсгалд record хадгалдаг distributed streaming platform гэж тодорхойлдог.
Эдгээр хэрэгслүүд нь “эвентэд суурилсан” уялдааг дэмжиж, сервисүүдийн хоорондын шууд хамаарлыг бууруулах боловч өгөгдлийн eventual consistency ба duplicate delivery зэрэг distributed системийн асуудлыг зохион байгуулах шаардлага үүсгэнэ.

Database-per-Service (өгөгдлийн эзэмшил)
microservices.io-ийн “Database per Service” паттерн нь микросервис бүрийн persistent өгөгдөл тухайн сервист “private” байх ёстой бөгөөд бусад сервисүүд зөвхөн API-гаар дамжин ашиглах ёстой гэж тодорхойлдог.
AWS whitepaper мөн микросервис бүр өөрийн өгөгдлөө хадгалж, энэ нь independence ба decentralization-ийг нэмэгдүүлдэг ч distributed data management-ийн шинэ сорилтуудыг үүсгэдэг гэж тайлбарладаг.
Eventual consistency (түр сааталтай нийцтэй байдал)
AWS-ийн тайлбарласнаар distributed системд consistency ба performance-ийн trade-off байдаг тул “instant updates” шаардахгүй, өгөгдлийн шинэчлэл багахан хугацаанд хоцрохыг (eventual consistency) зөвшөөрөх нь бодитой шийдэл болдог.
Distributed transaction ба Saga
Олон сервисийн өгөгдлийг нэг бизнес үйлдлээр шинэчлэх хэрэгцээ гарвал “нэг DB-ийн ACID transaction” шиг хийх боломж жирийн нөхцөлд байхгүй. microservices.io энэ нөхцөлд Saga-г “олон локал transaction”-ы дараалал болгон хэрэгжүүлж, алдаа гарвал compensation хийхийг зөвлөдөг.
Saga ойлголтын анхдагч судалгааны өгүүлэл болох нь “урт хугацааны transaction”-ийг жижиг transaction-уудын дараалал болгон задалж, шаардлагатай үед компенсэйшн үйлдлүүдээр буцаан сэргээх санааг танилцуулсан.

Мөн .NET-ийн гарын авлагад gateway нь reverse proxy байж, authentication, SSL termination, caching зэрэг боломжууд өгч болох боловч хэт төвлөрсөн “монолит aggregator” болох эрсдэлтэйг анхааруулсан.

Kubernetes нь кластерт Service болон Pod-д DNS record үүсгэж, consistent DNS name ашиглан сервисийг нээх боломж олгодог гэж албан баримт бичигтээ заасан.

Circuit Breaker-ийн санаа нь “алдаа/timeout” ихсэхэд хамгаалагдсан дуудлагыг таслан “fail fast” хийх замаар cascading failure-ээс сэргийлэх явдал гэж Fowler тайлбарласан.

Microsoft-ийн CQRS паттерн нь унших, бичих үйлдлийг тусдаа өгөгдлийн модел/хадгалалт болгон салгах нь performance, scalability, security-г сайжруулж болохыг тайлбарладаг.

Хөгжүүлэлт ба байршуулалт
Сервис тус бүрийг тусгаар release/deploy хийх боломж [Fowler–Lewis 2014].
CI/CD болон конфигурацийн төвөг өснө.
Масштаблалт
Зөвхөн хэрэгцээтэй сервисийг масштаблах
Data consistency ба coordination төвөг нэмэгдэнэ.
Найдвартай ажиллагаа
Fault isolation + resilience паттерн хэрэглэж чадвал тогтвортой байдал өснө
Cascading failure, timeout/retry storm зэрэг distributed failure mode гарч болно
Домэйн загварчлал
Bounded context дээр тулгуурласан тодорхой хараат бус байдлыг дэмжинэ
Boundary буруу тогтоовол “distributed monolith” болох эрсдэлтэй
Ажиглалт/оношилгоо
OTel, Prometheus, Jaeger гэх мэт стандарт хэрэгслээр end-to-end ойлголт авах боломж
Observability хэрэгсэлгүйгээр debugging бараг боломжгүй болдог
Сонголтын зөвлөмж
Микросервисийг сонгох шийдвэр нь “архитектурын тренд” бус, харин дараах нөхцөлүүд бодитоор биелж буй эсэх дээр тулгуурлах нь академик үндэслэлтэй.
Микросервис тохиромжтой нөхцөлүүд
Домэйн олон чадамжид салж, өөр өөрөөр өөрчлөгдөх/масштаблагдах магадлал өндөр (ж. эвент хайлт vs төлбөр vs мэдэгдэл).
DevOps ба автомат байршуулалт бодитоор хэрэгжих боломжтой (pipeline, container/orchestration).
Observability-ийн стек (traces/metrics/logs) төлөвлөгдөж, хэрэгжихээр байгаа.
6. Эрдэм шинжилгээний өгүүллүүдийн судалгаа
[A1] Performance comparison between a Monolithic and a Microservice Architecture (Bernsteiner et al., 2025)
Энэхүү судалгаа нь монолит ба микросервис архитектурын гүйцэтгэлийн ялгааг өндөр ачаалалтай үед хэмжиж харьцуулсан. Судлаачид ижил домэйн дээр прототип системүүдийг хэрэгжүүлж, ачааллын туршилтаар latency, throughput, error rate зэрэг үзүүлэлтүүдийг хэмжсэн байна. Судалгааны гол үр дүнгээр тодорхой тохиргоонд микросервис хувилбар нь хариу өгөх хугацааны хувьд 36% хурдан ажилласан бөгөөд алдааны түвшин мөн буурсан гэж дүгнэжээ. Энэ нь зөв зохион байгуулалттай микросервис архитектур өндөр ачааллыг илүү сайн даван туулж чадахыг харуулж байна. Миний дипломын ажилд уулзалт/эвент төлөвлөлтийн систем дээр гүйцэтгэлийн туршилт хийх аргачлал, хэмжилтийн логикийг тодорхойлоход энэхүү судалгааг загвар болгон ашиглах бөгөөд микросервис архитектурын давуу талыг батлах ишлэл болно.
[A2] From Monolith to Microservices: A Classification of Refactoring Approaches (Fritzsch et al., 2018/2019)
Энэхүү судалгаа нь монолитоос микросервис рүү шилжих рефакторинг аргачлалуудыг системтэйгээр ангилж, хөгжүүлэгчдэд зориулсан шийдвэрийн гарын авлагыг боловсруулсан. Судлаачид 10 гаруй рефакторинг хандлагыг гүнзгийрүүлэн шинжилж, задлалын техник, сервисийн ширхэглэл, шилжилтийн стратеги зэргээр ангилсан байна. Судалгааны гол дүгнэлтээр сервисийн ширхэглэл (granularity) сонголт нь шилжилтийн хамгийн хүндрэлтэй асуудал бөгөөд бүх арга нь тодорхой нөхцөл байдлаас хамааран хэрэгжих ёстой гэж үзжээ. Нэгдсэн стандарт арга байхгүй, харин контекстэд тохируулан сонгох шаардлагатай гэдгийг тогтоосон. Миний дипломын ажилд bounded context-д суурилсан сервис задлалыг зөвтгөх онолын тулгуур болох бөгөөд сервисүүдийг ямар зарчмаар хуваах, хэт жижигрүүлэх эрсдэлээс хэрхэн зайлсхийх талаар энэ судалгааны дүгнэлтүүдийг ашиглана.
[A3] Developing a Microservices Software System with Spring Cloud – A Case Study of Meeting Scheduler (ICEBE 2019, Springer, 2020)
Энэхүү судалгаа нь уулзалт төлөвлөлтийн (meeting planning) домэйн дээр Spring Cloud ашиглан микросервис системийг зохиомжлох, хэрэгжүүлэх туршлагыг тайлагнасан case study юм. Судлаачид сервисийн бүрэлдэхүүн, API gateway, service discovery зэрэг дэд бүтцийн бүрэлдэхүүн хэсгүүд болон хэрэгжилтийн зохион байгуулалтыг дэлгэрэнгүй тайлбарласан байна. Судалгааны үр дүнгээр Spring Cloud-ын экосистем нь сервис илрүүлэлт, чиглүүлэлт, тохиргооны удирдлага зэрэг түгээмэл асуудлуудыг шийдвэрлэхэд практик давуу талтай гэж дүгнэсэн. Миний дипломын ажилд системийн кейс сонголт (meeting/event planning) болон Spring Cloud-д суурилсан архитектурын компонентуудыг зөвтгөх гол ишлэл болох бөгөөд API gateway, service discovery зэрэг бүрэлдэхүүн хэсгүүдийг хэрхэн зохион байгуулах талаар энэ судалгааны загварыг дагаж мөрдөнө.
[A4] Authentication and authorization in microservice-based systems (Barabanov et al., 2020)
Энэхүү судалгаа нь микросервис системд баталгаажуулалт (authentication) ба эрхийн хяналт (authorization)-ын архитектурын хэв шинжүүдийг (patterns) тоймлон, практик зөвлөмж гаргасан. Судлаачид литературын тойм болон одоо байгаа стандартуудын дүн шинжилгээг хийж, микросервис орчинд аюулгүй байдлыг хангах янз бүрийн аргуудыг харьцуулан үзсэн байна. Судалгааны гол дүгнэлтээр API gateway дээр төвлөрсөн token-based механизм (OAuth2/JWT) болон сервисүүд дээр эрхийн хяналтыг хэрэгжүүлэх (policy enforcement) нь практик хэрэгжих боломжтой шийдэл гэж үзсэн. Мөн микросервис бүр бие даан баталгаажуулалт хийхээс илүүтэй gateway түвшинд нэгдсэн байдлаар шийдвэрлэх нь илүү үр дүнтэй болохыг тогтоосон. Миний дипломын ажилд JWT token ашиглан нэвтрэх эрх олгох, role-based access control (RBAC)-ийг хэрэгжүүлэх онолын суурь болох бөгөөд API gateway дээр authentication төвлөрүүлэх, сервисүүдэд authorization хэрэгжүүлэх загварыг энэ судалгаанд үндэслэн зохиомжлоно.
7. Microservices + уулзалт/эвент төлөвлөлтийн системтэй холбоотой судалгаануудын тойм
7.1 Meeting Scheduler монолитоос микросервис руу (ICEBE 2019, Spring Cloud)
Springer-ийн ICEBE 2019 эмхэтгэлд нийтлэгдсэн энэхүү кейс судалгаанд уулзалт төлөвлөлтийн системийг микросервис архитектурт шилжүүлэх туршлагыг тайлбарласан байна. Судлаачид системийг сервисүүд болгон задалж (service decomposition), Spring Cloud-ийн API gateway, service discovery зэрэг бүрэлдэхүүн хэсгүүдийг ашиглан түгээлт ба интеграцийг зохион байгуулсан гэж дурдсан. Судалгааны гол дүгнэлтээр "уулзалт төлөвлөлт" зэрэг домэйн нь сервисүүдээр задалж хэрэгжүүлэхэд тохиромжтой бөгөөд Spring Cloud экосистем нь микросервисийн нийтлэг асуудлуудыг (routing, discovery) хурдан шийдвэрлэх боломж олгодог болохыг харуулсан.
Миний дипломд авах сургамж: Сервисийн хил хязгаарыг домэйнээр тогтоох, gateway/discovery-г эхнээс нь төлөвлөх, monitoring/trace-ийг орхигдуулахгүй байх шаардлагатай. Миний ажил нь зөвхөн хэрэгжилт бус, харин монолит–SOA–microservices-ийн харьцуулалт болон гүйцэтгэлийн туршилтын хүрээг нэмэлтээр боловсруулна гэдгээрээ ялгаатай.
7.2 Klubraum апп (Akka, Cassandra, Kubernetes; 200,000+ хэрэглэгч; 99.9999% availability)
Klubraum нь Akka-д суурилсан түгээгдсэн микросервисүүд бүхий B2C эвент/нийгмийн апп бөгөөд Cassandra болон Kubernetes орчинд ажиллуулж, 200,000 гаруй хэрэглэгч, 10,000 гаруй групп, 99.9999% availability үзүүлэлттэй гэж үйлдвэрлэгчийн кейс тайланд дурдсан. "Six nines" availability (99.9999%)-д хүрэх нь өндөр өртөгтэй ч, зөв архитектур, тэсвэржилтийн загварууд шаарддагийг Akka-гийн тайлбар материал онцолсон байдаг.
Миний дипломд авах сургамж: Availability шаардлагыг дипломын түвшинд бодитой тавих (жишээ нь 99%), асинхрон мессеж ашиглах нь оргил ачаалалд тустай, observability-г хамгийн эхнээс суулгах нь чухал. Миний систем жижиг хүрээнд тул Kubernetes-ийг optional гэж үзэж, Docker Compose-оор хэрэгжилтийг төвлөрүүлэхээр төлөвлөж байна.
7.3 EventPlanner (LLM ашигласан, Telegram MiniApp хэлбэртэй систем)
EventPlanner нь Telegram Mini Apps (Telegram WebApps) платформ дээр хэрэгжүүлсэн эвент төлөвлөлтийн систем бөгөөд хэрэглэгчдэд Telegram дотор шууд ажиллах вэб апп ажиллуулах боломж олгодог. Mini app нь фронт-энд (Nuxt/React), API backend, auth/token механизм, notification интеграц гэсэн бүрэлдэхүүнтэй байж, Telegram event/callback-оор UI-тай холбогддог. Энэхүү систем нь LLM (YandexGPT) ашиглан эвент зохион байгуулахад туслах функцүүдийг хэрэгжүүлсэнээрээ онцлог юм.
Миний дипломд авах сургамж: Front-end-ийн төлөв ба auth-ыг gateway-тай нийцүүлэх, notification/interaction-ийг event-driven загвараар шийдэх боломжтойг харуулж байна. Миний дипломын зорилго LLM-ийг заавал багтаахгүй, харин архитектурын харьцуулалт ба microservices хэрэгжилтэд төвлөрөх тул энэ системийн архитектурын шийдлүүдийг сонгон авч хэрэгжүүлэх боломжтой.
7.4 Онлайн тасалбарын системийн судалгаа (36% илүү хурдан microservices)
Онлайн тасалбарын системийн прототип дээр монолит ба микросервис архитектуруудыг ачааллын нөхцөлд харьцуулсан судалгаагаар микросервис хувилбарын response time 36% хурдан байсан гэж тайлагнасан. Судлаачид throughput, error rate, resource usage зэрэг үзүүлэлтүүдийг ижил тест өгөгдөл дээр харьцуулан хэмжсэн байна. Энэхүү судалгаа нь зөв зохион байгуулалттай микросервис архитектур нь тодорхой нөхцөлд монолитоос илүү гүйцэтгэлтэй байж болохыг тоон үзүүлэлтээр баталсан чухал ажил юм.
Миний дипломд авах сургамж: Хэмжилтийн хүрээг (throughput, error rate, resource usage) эхнээс нь тогтоож, ижил тест өгөгдөл дээр харьцуулах шаардлагатай. Миний дипломын ажилд энэхүү судалгааны аргачлалыг дагаж, уулзалт төлөвлөлтийн систем дээр гүйцэтгэлийн харьцуулалт хийхдээ эдгээр үзүүлэлтүүдийг тусгах болно.

Microservices архитектурт ашиглах технологийн стек судалгаа
Орчин үеийн микросервис архитектур (microservices architecture) нь системийг жижиг, бие даасан сервисүүдэд хуваан хөгжүүлэх аргачлал бөгөөд энэ нь системийн өргөтгөх боломж (scalability), найдвартай байдал (reliability), болон уян хатан байдлыг (flexibility) сайжруулдаг гэж судлаачид үздэг [1]. Ийм архитектурыг хэрэгжүүлэхэд тохиромжтой технологийн стек сонгох нь системийн гүйцэтгэл болон хөгжүүлэлтийн үр ашигт чухал нөлөө үзүүлдэг.
1. Back-end технологи
1.1 Java / Spring Boot
Spring Boot нь Java програмчлалын хэл дээр суурилсан веб болон микросервис систем хөгжүүлэхэд ашиглагддаг фреймворк юм. Spring Boot нь auto-configuration, embedded server, болон production-ready features зэрэг боломжуудыг агуулдаг бөгөөд хөгжүүлэгчид богино хугацаанд REST API хөгжүүлэх боломжтой байдаг [2].
Мөн Spring Boot нь Spring Cloud экосистемтэй уялдан ажиллах боломжтой бөгөөд service discovery, configuration management, API gateway зэрэг микросервис архитектурт шаардлагатай бүрэлдэхүүнүүдийг хэрэгжүүлэхэд тохиромжтой байдаг [3].

SMTP, nodemailer
SMTP (Simple Mail Transfer Protocol) нь интернэт орчинд имэйл илгээхэд ашиглагддаг стандарт протокол юм. Энэ протокол нь клиентээс сервер рүү болон серверүүдийн хооронд имэйл дамжуулдаг бөгөөд өнөөгийн бүх имэйл системийн суурь технологи болно. SMTP протокол нь анх Internet Engineering Task Force (IETF)-ийн зүгээс тодорхойлогдож, RFC 5321 стандартаар албан ёсоор баталгаажсан. Уг стандартын дагуу SMTP нь текстэд суурилсан командууд ашиглан имэйл дамжуулалтыг гүйцэтгэдэг. SMTP нь өөрөө имэйл унших зориулалттай биш бөгөөд зөвхөн илгээх (send/transfer) үйлдлийг хариуцдаг. Харин имэйл уншихад POP3 эсвэл IMAP зэрэг бусад протоколууд ашиглагддаг.
Nodemailer нь Node.js орчинд SMTP протокол ашиглан имэйл илгээхэд зориулагдсан JavaScript сан юм. Энэхүү сан нь SMTP протокол дээр суурилан ажилладаг бөгөөд сервер талын програмуудаас имэйл илгээх процессыг хялбаршуулж өгдөг. Өөрөөр хэлбэл Nodemailer нь SMTP сервертэй холбогдож, authentication хийх, имэйл мессежийг форматлах (text, HTML), илгээх зэрэг бүх процессыг програмын түвшинд удирдах боломж олгодог.
EventMN системд SMTP болон Nodemailer-ийг ашиглан хэрэглэгчийн баталгаажуулалт (OTP), нууц үг сэргээх, 2FA зэрэг имэйлүүдийг автоматаар илгээдэг. Хэрэглэгч бүртгүүлэх болон нэвтрэх үед илгээгдэх нэг удаагийн код (OTP) нь SMTP-ээр дамжин имэйл серверт хүрч, Nodemailer-ийн тусламжтайгаар найдвартай илгээгддэг.
Эдгээр технологийг ашигласнаар имэйл илгээлтийн логик нь backend талд төвлөрч, хэрэглэгчийн аюулгүй байдал болон системийн найдвартай ажиллагаа хангагдсан.
Tailwind Tailwind CSS нь utility-first зарчимд суурилсан CSS framework бөгөөд урьдчилан тодорхойлогдсон utility class-уудыг ашиглан хэрэглэгчийн интерфейсийн загварыг шууд HTML болон JSX түвшинд удирдах боломж олгодог. Уламжлалт CSS-ээс ялгаатай нь стилийг тусдаа файлд бичих бус, компонентын түвшинд тодорхойлдог тул UI хөгжүүлэлт илүү хурдан, уян хатан болдог. Мөн Tailwind CSS нь responsive design, dark mode, theme customization зэрэг боломжуудыг анхнаасаа дэмждэг бөгөөд кодын давхардлыг багасгаж, дизайны тогтвортой байдлыг хангадаг. Иймээс Tailwind CSS нь орчин үеийн веб хөгжүүлэлтийн түгээмэл стандартын нэг болж байна.
neon
Neon нь cloud-д суурилсан serverless PostgreSQL өгөгдлийн сангийн үйлчилгээ юм. Энэ нь уламжлалт PostgreSQL серверийг байнга ажиллуулах шаардлагагүйгээр орчин үеийн serverless болон edge орчинд ашиглахад зориулагдсан. Өөрөөр хэлбэл, Neon нь PostgreSQL-ийн бүх үндсэн боломжийг хадгалсан боловч автомат масштаблалт, connection pool шаардлагагүй холболт, cloud-д ээлтэй архитектур-аараа ялгардаг.
Neon нь уламжлалт PostgreSQL-ээс дараах байдлаар ялгаатай:
Compute ба storage салангид архитектуртай
Өгөгдөл cloud storage-д хадгалагддаг
Query ирэх үед compute layer автоматаар идэвхждэг
Холболт TCP бус HTTP/WebSocket дээр суурилж ажилладаг
Энэ архитектур нь serverless орчинд түгээмэл тохиолддог connection limit, idle connection зэрэг асуудлыг шийддэг.
EventMN системд Neon-ийг дараах зорилгоор ашигласан:
PostgreSQL өгөгдлийн сан байрлуулах
Prisma ORM-тэй хамтран ашиглах
Serverless deployment (Vercel)-тэй уялдуулах
Prisma Neon Adapter-ийг ашигласнаар Next.js API route болон Server Components-оос өгөгдлийн сантай найдвартай, хурдан холбогдох боломж бүрдсэн.
Postgre sql
PostgreSQL нь SQL хэлийг ашигладаг, нээлттэй эхийн, өргөжүүлэх боломжтой, хүчирхэг өгөгдлийн сангийн систем юм. Энэ систем нь өгөгдлийг найдвартай хадгалах, удирдах, боловсруулахад зориулагдсан бөгөөд өндөр бат бөх чанар, найдвартай ажиллагаагаараа бусад өгөгдлийн сантай харьцуулахад давуу талтай. PostgreSQL нь ACID стандартыг бүрэн дэмждэг тул өгөгдлийн бүрэн бүтэн байдлыг сайн хангадаг. PostgreSQL нь дахин эмхэтгэх (recompile) шаардлагагүйгээр өөр програмчлалын хэл ашиглан функц, процедур бичих боломжийг олгодог. Энэ нь хөгжүүлэгчдэд системийг илүү уян хатан, хэрэглэгчийн шаардлагад нийцүүлэн өөрчлөх боломжийг олгодог. Мөн SQL өгөгдлийн төрөлтэй зэрэгцээд JSON, XML зэрэг харилцан холбоогүй өгөгдлийг хадгалах дэмжлэгтэй бөгөөд энэ нь PostgreSQL-ийг улам өргөн хэрэглээнд тохиромжтой болгодог.
shadcn.ui, radix.ui component
Shadcn UI болон Radix UI нь орчин үеийн React экосистемд зориулсан хэрэглэгчийн интерфейсийн шийдлүүд юм. Radix UI нь accessibility буюу хэрэглэгчдэд хүртээмжтэй байдлыг нэн тэргүүнд тавьсан headless UI primitive-үүдийг санал болгодог бол Shadcn UI нь эдгээр primitive-үүдийг Tailwind CSS-тэй хослуулан, олон хувилбартай (variant-driven) болон дахин ашиглагдах боломжтой UI компонентуудыг хялбар байдлаар бүтээхэд чиглэгддэг. Эдгээр сангуудыг ашигласнаар keyboard navigation, ARIA стандарт, polymorphic component зэрэг боломжууд автоматаар хангагдаж, тогтвортой бөгөөд хүртээмжтэй хэрэглэгчийн интерфейсийг богино хугацаанд хөгжүүлэх нөхцөл бүрддэг.

10.2 Use cases
Бүртгэл, нэвтрэх (register/login); Эвент үүсгэх (create event); Эвент засварлах (update event); Эвент устгах (delete event); Эвент хайх/шүүх (search/filter by date/location/category); Эвент дэлгэрэнгүй харах (view event details); Эвентэд бүртгүүлэх + суудал сонгох (book + seat selection); Бүртгэлээ цуцлах (cancel booking); Оролцогчдын жагсаалт харах (view participants); Мэдэгдэл хүлээн авах (receive notifications); Заалны суудлын мэдээлэл харах (view seating map).
10.3 Activity diagram-ын текстэн тайлбар (жишээ урсгал: эвентэд бүртгүүлэх)
Эхлэх (start) → хэрэглэгч эвент дэлгэрэнгүй нээнэ → “Бүртгүүлэх” товч дарна → систем нэвтэрсэн эсэхийг шалгана (auth check) → нэвтрээгүй бол login руу чиглүүлнэ → нэвтэрсэн бол суудлын мэдээлэл (seating) харуулна → хэрэглэгч суудал сонгоно → Booking Service захиалга үүсгэнэ → Event Service-ээс боломжит суудал/тоо баталгаажуулна (availability check) → амжилттай бол захиалга “CONFIRMED” болно → Notification Service мэдэгдэл илгээнэ → төгсөх (end).
10.4 Use case diagram-ын текстэн тайлбар
Actor: Organizer ↔ (Create/Update/Delete Event, View Participants, Send Announcements); Participant ↔ (Search/Filter, View Details, Book/Cancel, Receive Notifications); Admin ↔ (Manage Users, Moderate Events, Audit Logs). Системийн boundary нь “Meeting/Event Planning System”; Use case-үүд сервисүүдээр хуваагдан хэрэгжинэ (User/Event/Booking/Notification).

Тухайн систем нь өргөтгөх боломжтой (scalable) архитектуртай байх
Bounded Context зарчмаар microservices-үүдийг тодорхойлох
13.1 User Service (Хэрэглэгчийн контекст — User Context)
Үүрэг: Identity (identity), authentication (authentication), profile (profile), role (role) удирдана.
Гол функционал: register/login/JWT issue/profile update/admin list users.
DB загвар (PostgreSQL): users, roles (эсвэл users.role enum), refresh_tokens (optional).
API endpoints (жишээ): POST /api/users/register; POST /api/users/login; GET/PUT /api/users/profile; GET /api/users (admin).
Технологийн стек: Java + Spring Boot + Spring Security + PostgreSQL.
Яагаад тусдаа вэ (bounded context): Хэрэглэгчийн танин баталгаажуулалт ба профайл нь бусад домэйнээс тусдаа, өндөр хамгаалалттай контекст тул тусгаарлах нь зохимжтой.
13.2 Event Service (Эвентийн контекст — Event Context)
Үүрэг: Event lifecycle (event lifecycle), event metadata, category, search/filter.
Гол функционал: create/update/delete/list/search/details.
DB (PostgreSQL эсвэл MongoDB): events, categories, event_categories.
API: POST/GET/PUT/DELETE /api/events; GET /api/events/search …
Стек: Spring Boot + PostgreSQL/MongoDB.
Яагаад тусдаа вэ: Event өгөгдөл нь booking/notification-оос бие даасан, хайлт-шүүлт их тул scale-ийг тусад нь удирдах шаардлагатай.
13.3 Booking Service (Захиалгын контекст — Booking Context)
Үүрэг: Booking transaction, seat allocation, participant list.
Гол функционал: book/cancel/check/list.
DB (PostgreSQL): bookings, seats (optional), booking_audit (optional).
API: POST /api/bookings; DELETE /api/bookings/{id}; GET /api/bookings/event/{eventId}; GET /api/bookings/user/{userId}; GET /api/bookings/check…
Стек: Spring Boot + PostgreSQL + Redis (optional lock/cache).
Яагаад тусдаа вэ: Booking нь өндөр зэрэгцээ (high concurrency)-тэй “transactional” контекст тул тусдаа масштаблах/найдвартай болгох шаардлагатай.
13.4 Notification Service (Мэдэгдлийн контекст — Notification Context)
Үүрэг: Email/in-app мэдэгдэл, template, event queue consumer.
Гол функционал: send notification, consume booking/event events.
Харилцаа: Message broker (RabbitMQ/Kafka)-оор event-үүдийг авч боловсруулна (async).
Яагаад тусдаа вэ: Notification нь “ажлын дараалал” (background job) шинжтэй тул үндсэн request/response-оос салгах нь performance ба reliability-д тустай.
